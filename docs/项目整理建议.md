# 项目整理建议

本文档基于当前工程文件、`main.c`、`app_config.h`、`BSP/`、`Board/`、已有说明文档和 SysConfig 工程结构整理。目标是提高可维护性、减小源码体积，并为后续拆分 `main.c` 提供低风险路线。

## 当前结论

当前工程已经不是单文件小实验工程，而是一个包含 `Board/`、`BSP/`、控制算法、调试模式、赛题任务和数据记录工具的多模块工程。`main.c` 仍承担了过多职责，是当前维护成本最高的位置。

主要数据：

| 项目 | 当前情况 |
| --- | --- |
| `main.c` | 约 4400 行，约 150 KB，竞速日志已抽出 |
| `main.c` 函数数 | 仍偏多，后续可继续按任务模块拆分 |
| `race_log.h` | 约 879 行，竞速 RAM 日志系统，header-only 集成 |
| `app_debug_modes.h` | 约 728 行，包含 05/07 等静态内联调试逻辑，Task5 日志参数已结构化 |
| `app_config.h` | 327 行，约 280 个宏定义 |
| 已有可复用模块 | `Board/`、`BSP/`、`app_control.h`、`app_motion_utils.h`、`app_services.*`、`app_straight.h`、`app_task_ids.*`、`race_log.h` |
| 生成/构建文件 | `Debug/` 下 SysConfig 输出、对象文件、map/out 文件，不应手动修改 |

## 主要问题

### 0. 废弃判断不能只看文本引用

后续整理代码和宏定义时，不应只按“有没有被 `rg` 搜到”判断有效性。当前工程有不少旧调试函数和过渡函数，即使仍引用了某些宏，也可能已经不在当前任务入口中执行。

建议统一采用以下判断顺序：

| 顺序 | 判断项 | 说明 |
| --- | --- | --- |
| 1 | 当前编译配置 | 例如 `ENABLE_CONTEST_TASKS=1` 时，`main()` 默认进入 `run_task_dispatcher()` |
| 2 | 调度入口 | `run_task_dispatcher()` 当前可启动任务 1/2/3/4/5/6/7/10 |
| 3 | 任务调用链 | 从任务入口继续追踪实际会调用到哪些函数 |
| 4 | 参数实际作用 | 宏是否参与 PWM、航向、距离、停车、日志、条件编译或派生公式 |
| 5 | 分支可达性 | 函数会执行时，还要看该宏所在分支在当前任务参数/tag 下是否会进入 |

因此，后续文档中的“有效”应理解为“当前配置和任务路径下实际生效”，而不是“文本上被引用”。只在不可达旧函数中出现的宏，应标为“被引用但当前任务不生效”。

当前默认入口关系：

| 类别 | 入口 |
| --- | --- |
| 正式任务 | `run_task1_ab()`、`run_task2_abcd()`、`run_race_laps(1U)`、`run_race_laps(4U)` |
| 调试任务 | `run_motor_pid_stream()`、`run_task6_ac_c_turn_test()`、`run_motor_pd_stream()`、`run_task10_ab_zero_test()` |
| 公共服务 | `main()`、`run_task_dispatcher()`、`wait_task_uart_command()`、`GROUP1_IRQHandler()`、`UART_1_INST_IRQHandler()` |

### 1. `main.c` 职责过多

`main.c` 同时包含以下内容：

| 职责 | 示例 |
| --- | --- |
| 系统初始化和调度 | `main()`、`run_task_dispatcher()` |
| 任务编排 | `run_task1_ab()`、`run_task2_abcd()`、`run_race_laps()`、`run_task6_ac_c_turn_test()`、`run_task10_ab_zero_test()` |
| 通用运动段 | 直行到线、弧线循迹、快速转向、对准航向 |
| 数据记录 | Task2 RAM log、`race_log.h` 中的竞速 RAM log、实时 UART log |
| 中断入口 | `GROUP1_IRQHandler()`、`UART_1_INST_IRQHandler()` |

结果是：任何一个小参数或小任务修改仍需要在一个 4000 多行文件中定位。`race_log.h` 已经拆出一块大日志系统，但主文件仍需要继续按运动段和任务编排拆分。

### 2. 任务模块没有充分复用

工程中已经有可复用基础：

| 文件 | 当前职责 | 状态 |
| --- | --- | --- |
| `app_control.h` | `clamp_i32()`、`ramp_i32()`、直行 PID、航向滤波 | 可继续保留，建议移到 `app/control.*` |
| `app_straight.h` | 直行差速控制配置与输出计算 | 可继续保留，建议移到 `app/motion_straight.*` |
| `BSP/src/bsp_ir_tracking.c` | 八路红外 I2C 读取和解析 | 已相对清晰 |
| `BSP/src/bsp_tb6612.c` | 电机驱动 | 已相对清晰 |
| `BSP/inc/bsp_encoder.h` | 编码器运行接口和自检接口 | 接口较完整，但实现似乎主要在头文件内，后续可再迁移 |

但 `main.c` 中仍重复了很多运行循环：

| 重复模式 | 出现位置/现象 | 建议 |
| --- | --- | --- |
| `delay_ms_with_st011()` + `task_uart_stop_requested()` + 传感器采样 + PWM 输出 | 直行段、弧线段、Task11 线段、对准段 | 抽成“运动段运行框架”或至少抽公共 stop/tick 服务 |
| 编码器总距离计算 | 多处使用 `(abs_i32(B) + abs_i32(A)) / 2` | 抽 `motion_get_distance_count()` |
| JY62 航向读取和相对角计算 | 多处 `JY62_PeekNavigation()`、`normalize_cdeg()` | 抽 `nav_peek_yaw()`、`nav_yaw_progress()` |
| 红外有效线判断 | 多处判断 `ir_ok && !line_lost` | 抽 `ir_sample_line_valid()` |
| 丢线转向策略 | Task3/Task11 弧线与直线中相似 | 抽 `line_follow_calc_turn()` |
| RAM log 饱和转换 | Task2、Task5、Task11 都有 `sat_i16/u16` | 抽公共 `log_sat_i16()`、`log_sat_u16()` |

### 3. 函数名与实际用途不一致

部分函数名来自早期任务调试阶段，但实际已经被其它任务复用，导致阅读时容易误判。

| 当前函数名 | 实际用途 | 建议命名 |
| --- | --- | --- |
| `run_straight_to_line_segment()` | 已改为 Task1/Task2/Task11 复用的通用差速直行到线段 | 保留当前命名 |
| `run_task2_cd_exit_angle_straight()` | Task2 的 CD 段出口角直行封装，但内部使用 Task11 参数 | `task2_run_cd_straight_with_exit_yaw()` |
| `run_task2_da_race_arc()` | Task2 DA 段使用竞速弧线逻辑 | `task2_run_da_arc_segment()` |
| `race_diff_pid_reset()` | 差速 PID 初始化，不只属于竞速任务 | `diff_pid_reset()` |
| `race_drive_config()` | 差速直行/弧线驱动配置，不只属于竞速任务 | `diff_drive_config()` |
| `race_heading_turn_from_error()` | 通用航向误差转向量 | `heading_error_to_turn()` |
| `race_peek_yaw()` | JY62 航向读取封装 | `nav_peek_yaw()` |
| `task2_ram_enabled_for_tag()` | 根据 tag 判断 Task2 RAM log 段 | `task2_log_enabled_for_segment()` |
| `run_task10_ab_zero_test()` | AB 零角测试 | 可以保留，但建议移入 `task_debug.c` |

建议先改“外部不可见的 `static` 函数名”，这样风险低，不影响链接接口。

### 4. 调试代码与正式任务代码混在一起

当前有多类调试内容：

| 调试内容 | 位置 | 建议 |
| --- | --- | --- |
| Task5 RAM log | `app_debug_modes.h` 和 `main.c` 中共享存储 union | 拆到 `app/log_task5.*` 或保留但只在启用时编译 |
| Task2 RAM log | `main.c` 约 1300-1700 行附近 | 拆到 `app/log_task2.*` |
| 竞速 RAM log | 已抽到 `race_log.h` | 后续可再迁移为 `.c/.h`，但需要改 CCS 工程 |
| UART 实时打印 | `race_log_printf`、`race_event_printf` | 统一到 `app_log.h` |
| 轮速 PID/PD 调试 | `app_debug_modes.h`、`run_motor_pid_stream()`、`run_motor_pd_stream()` | 移到 `app/task_debug.*` |

调试日志代码体积大，而且大量格式化字符串会显著增加 flash 占用。建议默认关闭，必要时编译开关打开。

### 5. 配置宏过多且层次不清

`app_config.h` 当前把所有参数放在一个文件中：功能开关、硬件方向、通用控制、任务参数、调试参数、日志容量、经验参数都混在一起。

这会导致：

| 问题 | 影响 |
| --- | --- |
| 参数来源不清 | 不知道是硬件标定、算法参数、任务路线参数还是调试参数 |
| 派生参数和废弃参数混在一起 | 容易调错参数，或以为无效参数还能影响行为 |
| 任务间引用交叉 | 例如 Task11 参数依赖 Task3 参数，Task2 打印使用 Task11 参数 |
| 删除风险大 | 不知道宏是否只在条件编译分支使用 |

建议将配置拆成多个文件，或至少在 `app_config.h` 中重新分区。

## 分阶段整理路线

### 第一阶段：只整理，不改变行为

优先做不会改变运动逻辑的改动：

| 动作 | 目的 | 风险 |
| --- | --- | --- |
| 新建 `app_services.*` | 移出 ST011、延时服务、UART 停止命令、按键读取 | 低 |
| 新建 `app_task_ids.h` | 移出 `task_id_t` 和命令解析 | 低 |
| 新建 `app_motion_utils.h` | 移出 `normalize_cdeg()`、距离计算、红外有效判断 | 低 |
| 重命名明显不匹配的 `static` 函数 | 降低误读 | 低 |
| 把 `Task2/Task11 RAM log` 移入独立文件 | 减小 `main.c` 体积 | 中 |

验收标准：编译通过，串口启动提示和任务命令不变。

### 第二阶段：把任务拆文件

建议顺序：

| 顺序 | 模块 | 原因 |
| --- | --- | --- |
| 1 | `task_debug.c` | 调试任务与正式比赛路径耦合低 |
| 2 | `task1.c` | 主要是直行到线，逻辑相对简单 |
| 3 | `task2.c` | 依赖直行、弧线、Task2 log，拆前先抽公共段 |
| 4 | `task3.c` / `task4.c` | Task4 复用 Task3，适合一起整理 |
| 5 | `task11.c` | 参数和日志最多，最后拆最稳 |

验收标准：每拆一个任务，只验证对应任务编号能启动、能停止、关键串口日志字段不变。

### 第三阶段：压缩体积

建议重点：

| 目标 | 处理方式 |
| --- | --- |
| 减少 flash 字符串 | 将详细 `lc_printf()` 包到 `#if APP_VERBOSE_LOG_ENABLE` 或任务专用 log 开关 |
| 减少 RAM log 占用 | 默认关闭 `TASK2_RAM_LOG_ENABLE`、`TASK5_RAM_LOG_ENABLE`、`TASK11_RAM_LOG_ENABLE`，或只保留一个任务 log |
| 删除无引用宏 | 先删 `app_config.h` 中确认无引用的宏，再编译验证 |
| 删除过期调试路线 | 如果 06/10/11 已不再使用，移入 `task_debug.c` 并用总开关控制 |
| 合并相似打印函数 | 统一 stop reason、line sample、nav sample 的打印格式 |

### 其他废弃代码清理（2026-06-08 新增）

以下代码不属于"不可达函数"范畴，但同样是旧例程残留或重复接口，可一并清理。

#### `board.h` 中的 `u8/u16/u32/u64` 类型别名

```c
#ifndef u8
#define u8 uint8_t
#endif
// ... u16, u32, u64 同理
```

这些宏在当前代码中**没有被使用**（所有代码都用 `uint8_t` 等标准类型），是旧例程残留，2026-06-10 已删除。

#### `board.c` 中的重复延时函数

```c
void delay_1us(int __us) { delay_cycles( (CPUCLK_FREQ / 1000 / 1000)*__us); }
void delay_1ms(int __ms) { delay_cycles( (CPUCLK_FREQ / 1000)*__ms); }
```

`delay_1us` 和 `delay_1ms` 与 `delay_us` 和 `delay_ms` 实现完全相同，是重复接口。当前代码中只有 `delay_ms` 和 `delay_us` 被使用。2026-06-10 已从 `board.h` 和 `board.c` 中删除。

#### TB6612 旧例程兼容接口

```c
void TB6612_Motor_Stop(void);
void AO_Control(uint8_t dir, uint32_t speed);
void BO_Control(uint8_t dir, uint32_t speed);
```

这三个函数是旧例程兼容接口，当前代码中**没有任何调用者**（0 callers）。2026-06-10 已从 `bsp_tb6612.h` 和 `bsp_tb6612.c` 中删除。

#### `LOG_D` 宏和 `LOG_Debug_Out` 函数

当前代码中**没有使用 `LOG_D`**，所有打印都用 `lc_printf`。2026-06-10 已从 `board.h` 和 `board.c` 中删除。

#### `app_config.h` 中文档与代码不一致的宏

`docs/当前调试状态与后续问题.md` 中提到了 `TASK4_AC_RELATIVE_TURN_CDEG`，但 `app_config.h` 中实际没有定义这个宏。当前 Task4 的 AC 航向目标是直接使用 `TASK4_AC_HEADING_TARGET_CDEG`（等于 `TASK3_AC_HEADING_TARGET_CDEG`），不是相对转角。调试文档需要修正。

## 命名规范建议

建议统一以下规则：

| 类型 | 前缀 | 示例 |
| --- | --- | --- |
| BSP 驱动 | `BSP` 或模块名 | `IRTracking_ReadSample()`、`TB6612_SetDifferential()` |
| Board 工具 | `board_` 或继续保留现有 `lc_` | `lc_printf()` |
| 通用运动 | `motion_` | `motion_run_straight_until_line()` |
| 导航工具 | `nav_` | `nav_peek_yaw()` |
| 红外循迹控制 | `line_` | `line_calc_turn()` |
| 任务入口 | `taskN_run_` | `task2_run_abcd()` |
| 任务内部段 | `taskN_run_<segment>` | `task2_run_da_arc()` |

不建议继续使用 `task11_*` 表示通用函数，也不建议用 `task5_*` 表示通用直行 PID。

## 风险点

| 风险 | 说明 | 降低方式 |
| --- | --- | --- |
| 中断函数名不能随意改 | `GROUP1_IRQHandler`、`UART_1_INST_IRQHandler` 由启动文件/生成配置依赖 | 只移动文件，不改函数名 |
| SysConfig 生成文件不能手改 | `Debug/ti_msp_dl_config.*` 是生成文件 | 只改 `main.syscfg` 和应用代码 |
| 宏开关影响编译路径 | 一些函数只在 `#if` 分支中出现 | 每次删除宏后全量编译 |
| 任务参数互相引用 | 当前竞速配置宏仍使用 `TASK11_*` 命名，Task2 部分封装也引用这些参数 | 拆分前先画参数依赖表 |
| 运动逻辑对时序敏感 | 控制周期、延时和日志打印会影响实车 | 先做文件拆分，不改控制算法 |
