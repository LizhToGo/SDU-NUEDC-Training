# 串口与跑车流程

本文描述当前验收版固件。历史调试任务已经删除，UART0 只保留任务一二三四和运行中停车。

## 串口命令

| 命令 | 功能 | 入口 |
|---|---|---|
| `01` | 任务一：A -> B 直线 | `run_task1_ab()` |
| `02` | 任务二：A -> B -> C -> D -> A | `run_task2_abcd()` |
| `03` | 任务三：竞速路线 1 圈 | `run_race_laps(1U)` |
| `04` | 任务四：竞速路线 4 圈 | `run_race_laps(4U)` |
| `00` | 运行中强制停车 | `TASK_ID_STOP` |

命令同时支持二进制字节和 ASCII 文本。例如发送 HEX `03` 或文本 `03` 都能启动任务三。

当前验收版不接受 `05/06/07/08/10`。如果发送这些旧命令，解析器不会启动任务。

## 按键入口

| 按键 | 任务 |
|---|---|
| `A26` | 任务一 |
| `A24` | 任务二 |
| `B24` | 任务三 |
| `A22` | 任务四 |

任务三/四启动时会通过 `st011_start_pulse(RACE_START_ALARM_MS)` 非阻塞触发一次声光提示。

## 上车流程

1. 烧录后上电，等待启动信息和 `TASK ready`。
2. 确认 ready 文案为 `UART0 HEX bytes 01..04`。
3. 按实体按键或通过 UART0 发送 `01` 到 `04`。
4. 如需中止运行，发送 `00`。

任务三/四现在以起步时的 JY62 相对零点为航向基准，不再需要单独执行 Task10。摆车时仍需尽量保持预期初始朝向，避免零点本身带入人为误差。

## 日志开关

| 宏 | 作用 |
|---|---|
| `TASK2_RAM_LOG_ENABLE` | 任务二 RAM 日志 |
| `RACE_UART_LOG_ENABLE` | 任务三/四实时串口日志 |
| `RACE_RAM_LOG_ENABLE` | 任务三/四 RAM dump |

高速跑任务四时，优先使用 RAM dump；实时串口日志过密可能影响控制周期。

## 常见日志

| 日志 | 含义 |
|---|---|
| `TASK UART command accepted: id=N` | UART 命令已被接收 |
| `RACE_RAM_BEGIN` | 任务三/四 RAM dump 开始 |
| `RACE_CFG` | 当前竞速参数摘要 |
| `RACE_SEG` / `RACE_SEG_LINE` / `RACE_SEG_YAW` / `RACE_SEG_NAV` | 分段摘要 |
| `reason=point` | 正常点位触发 |
| `reason=force` | 距离保护触发 |
| `reason=uart_stop` | 串口停车 |

## 日志整理

工具脚本位于：

```text
tools/task11_log_to_csv.py
```

文件名保留历史名称，但可以处理当前 `RACE_*` RAM 日志。默认把结果写到 `tools/data/`，这些 CSV/文本输出是本地测试产物，不建议提交。
