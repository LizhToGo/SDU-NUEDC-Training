# 串口调试与跑车流程

本文档记录当前稳定版如何通过串口观察小车状态，以及如何根据日志判断任务一是否跑得正常。当前调试串口为 UART0，`115200 8N1`。

> 当前准备 push 的任务二调试状态、关键参数和后续问题，见 [当前调试状态与后续问题.md](当前调试状态与后续问题.md)。这一版空闲等待时不再持续输出 `JY62 mode=idle`，收到任务命令后开始输出任务数据，任务结束后停止周期输出。

## 1. 串口配置

UART0 由 `main.syscfg` 配置：

| 项目 | 当前值 |
|---|---|
| 串口实例 | `UART_0` |
| 发送引脚 | `PA28 / UART0_TX` |
| 接收引脚 | `PA31 / UART0_RX` |
| 波特率 | `115200` |
| 数据格式 | 8 数据位，无校验，1 停止位 |

接线时交叉连接：

| MSPM0G3507 | 串口模块 |
|---|---|
| `PA28 / TX` | 模块 `RX` |
| `PA31 / RX` | 模块 `TX` |
| `GND` | 模块 `GND` |

如果同时接 DAPLink 串口和蓝牙串口，不要让两个模块的 `TX` 同时直接驱动 MSPM0 的 `PA31/RX`。最稳妥的方式是同一时间只接一个发送源。

## 1.1 当前串口命令

| 命令 | 作用 |
|---|---|
| `01` | 启动任务一 A -> B 直线 |
| `02` | 启动任务二 A -> B -> C -> D -> A |
| `03` | 进入任务三入口，目前路线逻辑未实现 |
| `04` | 进入任务四入口，目前路线逻辑未实现 |
| `05` | 启动左右轮编码器/PWM 测试 |
| `00` | 强制停车 |

任务命令被接受后会打印：

```text
TASK UART command accepted: id=...
```

`05` 模式下可用 `00` 停止，停止时会打印：

```text
PID motor stream stop: UART0 00
```

## 2. 当前任务一流程

当前默认开关在 `app_config.h`：

```c
#define ENABLE_CONTEST_TASKS         (1)
#define ENABLE_IR_TRACKING_UART_TEST (0)
#define ENABLE_LINE_FOLLOW_TEST      (0)
#define ENABLE_JY62_NAV              (1)
```

上电后会打印 BOOT 信息，然后进入任务等待：

```text
BOOT: UART OK, IR line follow firmware
BOOT: JY62 UART1 ready, PA08 TX -> RXD, PA09 RX <- TXD, baud=115200
BOOT: encoder state loaded, B=PA14/PA15 A=PA16/PA17
BOOT: TB6612 ready, A motor and B motor enabled
TASK ready: A26/UART0 01=task1, A24/UART0 02=task2, B24/UART0 03=task3, A22/UART0 04=task4, UART0 05=PWM test, UART0 00=stop
```

操作流程：

1. 把小车手动摆在 A 点，车头朝 B 点。
2. 打开串口工具，确认看到 `TASK ready`。
3. 按下 A26 任务一按键，或从 UART0 发送 `01`。
4. 声光模块响一次，程序等待一小段时间，JY62 当前航向置零。
5. 小车开始任务一 A->B 直线段。
6. 到 B 点附近，红外模块识别到黑线后刹车，声光模块再响一次。

四个实体按键映射如下：

| 按键引脚 | 任务 |
|---|---|
| `A26 / PA26` | 任务一 |
| `A24 / PA24` | 任务二 |
| `B24 / PB24` | 任务三，目前占位 |
| `A22 / PA22` | 任务四，目前占位 |

任务开始时会重置直行 PID、航向滤波、编码器距离计数。任务结束后不会额外清零编码器总数；下一次任务开始会再次清零，所以分析单次任务时看该次 `TASK1` 和 `TASK1 stop` 即可。

## 3. `JY62` 日志

空闲、上电置零、任务开始置零、任务运行时都会输出类似日志：

```text
JY62 mode=task1 t=1000 df=198 ok=1 flags=0x07 yaw_cdeg=-157 rel_cdeg=-138 gz_mdps=3479 gzlp_mdps=976 rx=17334 head=1587 frames=1575 err=0/0/0
```

| 字段 | 含义 |
|---|---|
| `mode` | 当前打印场景：`boot_zero`、`idle`、`task1_zero`、`task1` |
| `t` | 当前场景下的时间，单位 ms |
| `df` | 距离上次读取新增的 JY62 有效帧数 |
| `ok` | JY62 导航数据是否有效，1 表示有效 |
| `flags` | 最近更新的数据类型标志，常见 `0x07` 表示角度、角速度等都有更新 |
| `yaw_cdeg` | JY62 当前 yaw，单位为 0.01 度 |
| `rel_cdeg` | 相对最近一次置零的 yaw，单位为 0.01 度 |
| `gz_mdps` | Z 轴角速度，单位 mdps |
| `gzlp_mdps` | 低通滤波后的 Z 轴角速度，单位 mdps |
| `rx` | UART1 收到的总字节数 |
| `head` | 成功识别到帧头的次数 |
| `frames` | 成功解析的帧数 |
| `err` | `校验错误/UART错误/溢出` 计数 |

判断方法：

| 现象 | 说明 |
|---|---|
| `ok=1` 且 `err=0/0/0` | JY62 通信正常 |
| `rel_cdeg` 在 `task1_zero` 后接近 0 | 任务开始置零成功 |
| 空闲时手动晃车，`gzlp_mdps` 变大 | 陀螺仪能看到车身晃动 |
| `ok=0` 或 `frames` 不增长 | 优先查 JY62 供电、共地、TX/RX 是否交叉 |

## 4. `TASK1` 周期日志

任务一运行时每 `TASK1_REPORT_PERIOD_MS` 打印一次长日志。串口工具可能会把一行视觉上折成两行，但固件本身输出的是同一条 `TASK1` 记录。

示例：

```text
TASK1 t=1200 dist=6475 arm=1 raw=0xFF mask=0x00 cnt=0 lost=1 ir=1 nav=1 h_raw=-110 h_flt=-123 h_use=-123 h_gain=100 h_wob=0 gzlp=1770 h_corr=4 hp=0 B_total=6491 A_total=6459 d_err=32 d_corr=1 B_spd=94 A_spd=95 v_err=-1 P=-22 I=64 D=-18 v_corr=1 bal=2 corr=6 base=485/525 B_pwm=479 A_pwm=531
```

### 基本状态字段

| 字段 | 含义 |
|---|---|
| `t` | 任务开始后的时间，单位 ms |
| `dist` | 当前距离估计，`(abs(B_total)+abs(A_total))/2` |
| `arm` | 是否允许红外黑线触发停车，`dist >= TASK1_B_LINE_ARM_COUNT` 后为 1 |
| `raw` | 红外模块原始字节，通常 `0xFF` 表示没有看到黑线 |
| `mask` | 转换后的黑线掩码，`bit0=X1`，`bit7=X8`，1 表示对应探头看到黑线 |
| `cnt` | 当前看到黑线的红外探头数量 |
| `lost` | 1 表示当前没有看到黑线 |
| `ir` | 红外 I2C 读取是否成功，1 成功，0 失败 |
| `nav` | JY62 导航数据是否有效，1 有效 |

### 航向修正字段

| 字段 | 含义 |
|---|---|
| `h_raw` | JY62 原始相对 yaw，单位 0.01 度 |
| `h_flt` | 一阶低通后的航向误差 |
| `h_use` | 经过死区和晃动降权后实际用于控制的航向误差 |
| `h_gain` | 根据 `gzlp` 计算出的航向修正增益百分比，100 表示不降权 |
| `h_wob` | 晃动标志，1 表示角速度较大，正在降低航向修正权重 |
| `gzlp` | 低通后的 Z 轴角速度，单位 mdps |
| `h_corr` | JY62 航向修正量 |
| `hp` | 航向优先标志，1 表示在特定条件下压掉编码器平衡修正，让航向修正优先生效 |

航向经验公式在 `app_control.h` 中。核心逻辑是：`rel_cdeg` 先低通，角速度大时进一步降权，小误差进入死区，最后再按 `TASK1_HEADING_CORR_DIVISOR` 转成 `h_corr`。

### 编码器和 PID 字段

| 字段 | 含义 |
|---|---|
| `B_total` | 左轮 B 电机任务内累计编码器计数 |
| `A_total` | 右轮 A 电机任务内累计编码器计数 |
| `d_err` | `B_total - A_total`，用于左右轮总距离修正 |
| `d_corr` | 距离累计误差产生的修正量 |
| `B_spd` | 当前控制周期内左轮速度估计，`abs(B_delta)` |
| `A_spd` | 当前控制周期内右轮速度估计，`abs(A_delta)` |
| `v_err` | 速度 PID 误差，`B_spd - A_spd` |
| `P/I/D` | 速度 PID 三项的未缩放中间值 |
| `v_corr` | 速度 PID 输出的修正量 |
| `bal` | 编码器平衡修正，约等于 `v_corr + d_corr` |

`v_err > 0` 表示左轮 B 当前更快；`v_err < 0` 表示右轮 A 当前更快。

### PWM 字段

| 字段 | 含义 |
|---|---|
| `corr` | 最终修正量，综合 `bal` 和 `h_corr` 后再限幅 |
| `base` | 起步斜坡后的基础 PWM，格式为 `B基础/A基础` |
| `B_pwm` | 最终给左轮 B 电机的 PWM |
| `A_pwm` | 最终给右轮 A 电机的 PWM |

`corr` 为正时，程序降低 B 轮 PWM、提高 A 轮 PWM；`corr` 为负时，程序提高 B 轮 PWM、降低 A 轮 PWM。

## 5. `TASK1 stop` 结束日志

任务一结束时会输出一条停止日志：

```text
TASK1 stop: reason=line t=1280 dist=6969 arm=6000 force=9500 raw=0xDF mask=0x04 cnt=1 lost=0 ir=1 nav=1 rel_cdeg=-118 B_total=6985 A_total=6954
```

| 字段 | 含义 |
|---|---|
| `reason` | 停车原因：`line` 黑线停车，`force` 编码器距离兜底停车，`timeout` 超时停车 |
| `t` | 任务运行时间，单位 ms |
| `dist` | 停车时距离估计 |
| `arm` | 红外停车允许阈值，即 `TASK1_B_LINE_ARM_COUNT` |
| `force` | 强制停车阈值，即 `TASK1_FORCE_STOP_COUNT` |
| `raw/mask/cnt/lost/ir` | 停车瞬间最后一次红外采样状态 |
| `nav` | 停车瞬间 JY62 数据是否有效 |
| `rel_cdeg` | 停车瞬间相对任务开始的 yaw，单位 0.01 度 |
| `B_total/A_total` | 停车瞬间左右轮任务内累计编码器计数 |

正常任务一优先希望看到：

```text
reason=line ir=1 cnt>=1 lost=0
```

如果出现 `reason=force`，说明程序没有在允许停车距离之后识别到 B 点黑线，只能靠编码器距离兜底停车。

## 6. 怎么判断一组日志好不好

| 日志现象 | 可能含义 | 建议处理 |
|---|---|---|
| `reason=line`，`dist` 稳定在同一段范围 | B 点黑线识别稳定 | 当前参数可作为留档 |
| `reason=force`，且运行中 `ir=0` | 红外 I2C 读取失败 | 查红外模块供电、共地、SDA/SCL、连接松动 |
| `cnt>0` 但 `arm=0` | 太早看到线，程序按设计忽略 | 如果确实是 B 点线，降低 `TASK1_B_LINE_ARM_COUNT` |
| `arm=1` 后很久仍 `lost=1` | 到 B 点附近后没看到黑线 | 查红外高度、阈值、电池电压、停车阈值 |
| `B_total` 和 `A_total` 长期差距快速变大 | 左右轮累计距离不平衡 | 查轮胎打滑、编码器线、电机线，必要时调 `d_corr` |
| `B_spd` 或 `A_spd` 长期接近 0 | 某侧编码器或电机异常 | 架空检查电机、编码器、TB6612 接线 |
| `corr` 长期接近 `STRAIGHT_CORR_MAX` | 控制量打满 | 先查硬件，再考虑降低速度或调 PID |
| `h_wob=1` 经常出现但车身没有晃 | JY62 角速度噪声或固定不牢 | 检查 JY62 固定、供电、线束 |
| `rel_cdeg` 逐渐变大但 `h_corr` 很小 | 航向权重偏低或死区偏大 | 适当增强 JY62 权重，但要防止起步误修正 |

当前几次较好的任务一日志特征：

- `reason=line`。
- `t` 约在 `1280ms` 到 `1320ms`。
- `dist` 约在 `6900` 到 `7250`。
- `ir=1`，停车时 `cnt>=1`。
- `B_total` 和 `A_total` 差距较小。
- `corr` 没有长期贴近限幅。

## 7. 保留的调试模式

### 红外模块串口测试

把 `app_config.h` 中开关改为：

```c
#define ENABLE_CONTEST_TASKS         (0)
#define ENABLE_IR_TRACKING_UART_TEST (1)
```

典型输出：

```text
IR t=100 raw=0xE7 mask=0x18 count=2 lost=0 pos=0 err=0
```

字段含义：

| 字段 | 含义 |
|---|---|
| `raw` | 模块原始字节，`bit7=X1`，`bit0=X8` |
| `mask` | 程序转换后的黑线掩码，`bit0=X1`，`bit7=X8`，1 表示黑线 |
| `count` | 当前看到黑线的探头数量 |
| `lost` | 1 表示 8 路都没有看到黑线 |
| `pos` | 加权线位置，中心为 0，左负右正 |
| `err` | 循迹误差，当前等于 `pos` |

### 基础直行 PID 测试

关闭正式任务和其他调试模式后，会进入基础直行 PID：

```c
#define ENABLE_CONTEST_TASKS         (0)
#define ENABLE_IR_TRACKING_UART_TEST (0)
#define ENABLE_LINE_FOLLOW_TEST      (0)
```

典型输出：

```text
PID t=1000 B_cnt=56 A_cnt=55 B_spd=56 A_spd=55 err=1 P=16 I=-65 D=0 corr=-2 B_pwm=302 A_pwm=308
```

这个模式只用于架空或低速验证左右轮速度闭环，不包含红外停车和 JY62 航向修正。

### 纯红外循迹测试

把 `ENABLE_LINE_FOLLOW_TEST` 改为 1 可以进入纯红外循迹调试。该模式主要用于后续任务二/弧线段，不是当前任务一稳定版的默认路径。

## 8. 烧录前检查清单

每次烧录前建议快速检查：

- `app_config.h` 里的模式开关是否是你想要的组合。
- 小车是否架空，或者电机电源是否暂时断开。
- TB6612 的 `VM` 电机电源和主控 `GND` 是否共地。
- JY62 是否固定牢，线束是否会拖动模块。
- 八路红外模块是否已校准，离地高度是否稳定。
- 串口工具是否设置为 `115200 8N1`。
- 新增 `.c` 文件后是否在 CCS 里执行过 `Refresh` 和 `Clean Project`。

## 9. 常见问题

### 串口没有输出

优先检查：

- 是否已经调用 `SYSCFG_DL_init()`。
- 串口工具波特率是否为 `115200`。
- `TX/RX` 是否接反。
- 是否有多个串口工具同时占用同一个端口。
- 蓝牙串口模块是否已经配对并打开正确的 COM 口。

### 红外读取失败

优先检查：

- `SDA=PB3`、`SCL=PB2` 是否接对。
- `VCC/GND` 是否接好。
- I2C 地址是否为 `0x12`。
- 模块 I2C 上拉电压是否超过 MSPM0 输入可接受范围。

### 小车突然明显转向或转圈

优先检查：

- A/B 电机线、编码器线是否松动。
- TB6612 `AIN1/AIN2/BIN1/BIN2` 是否接牢。
- `B_spd/A_spd` 是否有一侧长期为 0。
- `B_total/A_total` 是否有一侧不增长或方向异常。
- JY62 是否在按键置零后被手碰到，导致 `h_raw` 一开始就很大。

### 烧录时报 flash 验证错误

可以先尝试：

- 断电重插开发板。
- CCS 里做一次全片擦除或重新 Debug。
- 检查是否有程序正在占用调试器。
- 确认供电稳定，避免烧录过程中掉电。

如果仍然反复出现，再单独排查调试器、驱动和板卡连接。
