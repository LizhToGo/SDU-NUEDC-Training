# 项目实现思路

本文从软件结构、硬件抽象、控制算法和调试流程四个角度说明当前自动行驶小车固件的实现思路。当前工程基于 TI MSPM0G3507，使用 CCS、SysConfig 和 DriverLib，面向赛道任务的 A -> B -> C -> D -> A 自动行驶。

## 1. 总体目标

当前固件的目标不是单独验证某一个外设，而是把电机、编码器、红外循迹、JY62 航向和串口调试整合成一套可反复实车调参的任务框架。

核心思路：

- 直线段依靠编码器速度闭环、左右累计距离修正和 JY62 航向修正共同保持直线。
- 弧线段依靠八路红外循迹模块做 PD 循迹。
- 任务状态机负责 A/B/C/D 分段切换，每段只处理自己的控制目标。
- 串口日志尽量把每一层控制量都打印出来，方便根据实车现象定位问题。
- 参数集中放在 `app_config.h`，方便快速调试而不破坏控制逻辑。

## 2. 软件分层

当前项目主要分为四层：

| 层级 | 文件 | 作用 |
|---|---|---|
| SysConfig 配置层 | `main.syscfg` | 时钟、GPIO、PWM、UART、I2C、中断等外设配置源头 |
| BSP 驱动层 | `BSP/inc`、`BSP/src` | TB6612、电机编码器、八路红外、JY62 等硬件接口 |
| 控制辅助层 | `app_control.h`、`app_config.h` | 参数、限幅、斜坡、速度 PID、航向滤波 |
| 任务应用层 | `main.c`、`app_debug_modes.h` | 任务调度、直线段、弧线段、调试模式、串口输出 |

这种分层的好处是：底层硬件接线变化时优先改 `main.syscfg` 和 BSP；控制参数变化时优先改 `app_config.h`；赛道策略变化时主要改 `main.c`。

## 3. 硬件抽象

### 3.1 电机和车轮映射

实车自测确认：

| TB6612 通道 | 实际车轮 | PWM |
|---|---|---|
| A 电机 | 右轮 | `PA0 / PWMA` |
| B 电机 | 左轮 | `PA1 / PWMB` |

上层统一使用车体语义：

```c
TB6612_SetDifferential(left_speed, right_speed);
```

内部映射保持为：

```c
TB6612_SetMotor(TB6612_MOTOR_B, left_speed);
TB6612_SetMotor(TB6612_MOTOR_A, right_speed);
```

这样任务层只需要按“左轮、右轮”思考，不需要记住 TB6612 的 A/B 通道和实际车轮的关系。

### 3.2 编码器

编码器使用 GPIOA 中断读取 A/B 相：

| 车轮 | 编码器引脚 |
|---|---|
| 左轮 B | `PA14 / PA15` |
| 右轮 A | `PA16 / PA17` |

编码器驱动维护两类数据：

- 控制周期增量：用于计算当前速度 `B_spd / A_spd`。
- 任务内累计量：用于估算距离 `dist` 和左右累计误差 `d_err`。

每段任务开始时会调用 `encoder_reset_distance_counts()`，让当前段的距离从 0 开始算。

### 3.3 八路红外循迹模块

八路红外通过 I2C 读取数字量，驱动层会把原始字节转换成：

- `raw`：模块原始值。
- `mask`：黑线掩码，`bit0..bit7` 对应 `X1..X8`。
- `active_count`：当前看到黑线的探头数量。
- `position_error`：线相对传感器中心的位置误差。
- `line_lost`：是否丢线。

直线段主要用红外判断到点停车；弧线段用红外误差做循迹控制。

### 3.4 JY62 航向

JY62 使用 UART1 接收姿态帧。驱动解析角度帧和角速度帧，并提供：

- 当前 yaw：`yaw_cdeg`。
- 相对任务开始置零后的 yaw：`yaw_relative_cdeg`。
- Z 轴角速度：`gyro_z_mdps`。
- 低通后的 Z 轴角速度：`gyro_z_filtered_mdps`。
- 有效性标志：`valid`。

任务开始时调用 `JY62_SetYawZeroToCurrent()`，把当前车头方向作为 0 航向。直线段使用相对航向做纠偏。

## 4. 任务调度

正式模式由 `run_task_dispatcher()` 负责等待命令并启动任务。

当前 UART0 命令：

| 命令 | 功能 |
|---|---|
| `01` | 任务一 A -> B 直线 |
| `02` | 任务二 A -> B -> C -> D -> A |
| `05` | 左右轮编码器/PWM 测试 |
| `00` | 强制停车 |

任务二流程：

```text
TASK2_AB: A -> B 直线
TASK2_BC: B -> C 弧线循迹
TASK2_CD: C -> D 直线
TASK2_DA: D -> A 弧线循迹
```

分段实现的好处是每一段的控制目标很清晰：

- AB/CD 只关心直线、航向和到线停车。
- BC/DA 只关心红外循迹和线尾结束。
- 段与段之间重置编码器距离，避免上一段累计量影响下一段。

## 5. 直线段控制思路

直线段由 `run_straight_to_line_segment()` 实现，任务一、任务二 AB、任务二 CD 都复用这一套逻辑。

### 5.1 起步斜坡

为了避免一上电就给最终 PWM 导致打滑或冲击，直线段从起步 PWM 逐步过渡到目标 PWM：

```c
#define TASK1_RAMP_B_START_PWM    (500)
#define TASK1_RAMP_A_START_PWM    (540)
#define STRAIGHT_B_BASE_PWM       (510)
#define STRAIGHT_A_BASE_PWM       (560)
```

当前车体右轮 A 需要更高 PWM 才能和左轮 B 接近等效速度，所以 A 的起步和最终 PWM 都高于 B。

### 5.2 速度 PID

速度 PID 的目标是让左右轮当前周期速度相等：

```text
v_err = B_spd - A_spd
```

- `v_err > 0`：左轮 B 快，需要降低 B、提高 A。
- `v_err < 0`：右轮 A 快，需要提高 B、降低 A。

PID 输出为 `v_corr`，用于短时修正速度差。

### 5.3 累计距离修正

只看瞬时速度容易忽略长期偏差，所以直线段还会计算：

```text
d_err = B_total - A_total
```

如果某一侧长期多走，会通过 `d_corr` 纠回来。这样即使短时速度接近，也能抑制慢慢偏航。

### 5.4 JY62 航向修正

JY62 提供相对航向误差，控制层先做低通和死区处理，再转换成 `h_corr`。

为了减少车身起步晃动造成误判，航向滤波会根据 Z 轴角速度自动降权：

- 角速度小：保留航向修正。
- 角速度大：降低航向修正权重。
- 误差小于死区：不修正。

最终直线段修正量近似为：

```text
corr = v_corr + d_corr + h_corr
```

再通过限幅后作用到左右轮：

```text
B_pwm = base_b_pwm - corr
A_pwm = base_a_pwm + corr
```

### 5.5 到点停车

直线段不会一看到黑线就停车，而是先等待距离达到允许阈值：

```c
#define TASK1_B_LINE_ARM_COUNT (6000)
```

超过该距离后，只要红外检测到黑线探头数量达到要求，就认为到达目标点并刹车。如果始终没识别到黑线，会用强制距离兜底：

```c
#define TASK1_FORCE_STOP_COUNT (9500)
```

## 6. 弧线循迹思路

弧线段由 `run_arc_line_follow_segment()` 实现，任务二 BC 和 DA 复用这一套逻辑。

### 6.1 红外误差

弧线段周期读取八路红外，得到当前黑线位置误差：

```text
raw_err -> filtered_error
```

如果误差很小，会进入死区，避免小范围来回抖动。

### 6.2 PD 转向

弧线转向量主要由红外误差和误差变化率决定：

```text
turn_raw = filtered_error / KP_DIVISOR + derivative / KD_DIVISOR
```

再经过：

- D 项限幅。
- 总转向限幅。
- 转向变化率限制。

这样可以减轻红外误差跳变带来的突然猛打方向。

### 6.3 动态降速

弧线基础速度不是固定不变的。误差越大、D 项越大，基础速度越低：

```text
base_pwm = TASK2_ARC_BASE_PWM - error_slowdown - derivative_slowdown
```

再限制到 `TASK2_ARC_MIN_BASE_PWM..TASK2_ARC_BASE_PWM` 范围内。

最终输出：

```text
left_pwm  = base_pwm + turn
right_pwm = base_pwm - turn
```

### 6.4 当前弧线入口问题

当前版本的主要已知问题是：刚进入弧线的一瞬间容易冲出赛道。

当前代码状态：

- 没有启用旧版强制转向。
- 没有合并其他 branch 中的数学前馈逻辑。
- 弧线中后段如果红外能抓住线，PD 循迹可以继续工作。

后续更推荐把改动集中在“入弧最初一小段”：

- 短时间、小幅度的入弧预转。
- 或合并数学前馈，依据入 B/D 时红外位置和几何关系给初始方向补偿。
- 一旦 `cnt>=1`，立即交还给红外 PD，避免长期强制转向破坏循迹。

## 7. 调试模式设计

`app_debug_modes.h` 保留了几个独立调试入口，方便把问题拆开查：

| 模式 | 用途 |
|---|---|
| `05` 轮速测试 | 单独观察 B/A 编码器速度、PID 修正和最终 PWM |
| 红外 UART 测试 | 单独查看八路红外原始值、黑线掩码和位置误差 |
| 纯红外循迹测试 | 不跑完整任务，只测红外循迹控制 |
| 编码器自检 | 架空车体时检查左右编码器是否有脉冲 |

调车时优先拆分问题：

- 直线偏：先跑 `05` 看轮速和编码器。
- 识别不到线：先跑红外测试看 `raw/mask/cnt`。
- 航向异常：看 `JY62` 的 `ok/frames/rel_cdeg/gzlp_mdps`。
- 弧线失控：看 `TASK2_BC/DA` 的 `raw_err/err/der/base/turn/L/R`。

## 8. 串口日志设计

当前串口输出不是只打印“跑到哪里”，而是把控制链路中的中间量都打出来。

直线段重点字段：

| 字段 | 含义 |
|---|---|
| `dist` | 当前段距离估计 |
| `raw/mask/cnt/lost` | 红外状态 |
| `h_raw/h_flt/h_use/h_corr` | 航向误差和航向修正 |
| `B_total/A_total` | 左右轮累计计数 |
| `B_spd/A_spd` | 左右轮当前速度 |
| `v_err/P/I/D/v_corr` | 速度 PID 状态 |
| `d_err/d_corr` | 累计距离修正 |
| `corr` | 最终综合修正 |
| `base/B_pwm/A_pwm` | 基础 PWM 和最终 PWM |

弧线段重点字段：

| 字段 | 含义 |
|---|---|
| `yaw_rel/yaw_abs/yaw_err` | 当前相对航向和目标差 |
| `raw/mask/cnt` | 红外状态 |
| `raw_err/err/der` | 红外误差、滤波误差和 D 项 |
| `base` | 动态基础速度 |
| `turn/turn_raw` | 转向输出 |
| `L/R` | 最终左右轮 PWM |

通过这些字段可以把实车现象反推到具体控制原因，而不是只凭肉眼猜。

## 9. 安全和兜底策略

当前固件保留了几个兜底保护：

- UART0 输入 `00` 可强制停车。
- 直线段有最大运行时间和强制距离停车。
- 弧线段有最大运行时间和强制距离停车。
- 弧线丢线后会根据距离、丢线次数和航向窗口判断是否结束。
- 每段任务失败会打印 `abort after ...`，避免静默进入下一段。

这些保护的目标是：宁可中止，也不要在失去定位后继续盲跑。

## 10. 后续优化方向

后续建议按风险从小到大推进：

1. 保留当前能跑版本，先只调参数，不大改逻辑。
2. 用 `05` 和任务日志确认直线轮速参数是否稳定。
3. 针对入弧瞬间冲出赛道，先降低入口速度或增加短时预转。
4. 合并 branch 中的数学前馈逻辑，但必须先核对修正方向。
5. 给前馈和入弧辅助增加串口字段，确认它只在入弧阶段生效。
6. 若弧线中段仍摆动，再调 `TASK2_ARC_KP_DIVISOR`、`TASK2_ARC_KD_DIVISOR`、`TASK2_ARC_TURN_SLEW_STEP` 和动态降速参数。

总体原则：先保证每一段可解释，再追求速度。只要日志能解释实车动作，后续调参就不会失控。
