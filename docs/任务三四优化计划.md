# 任务三/四竞速优化正式计划

版本：v1.0  
范围：任务三、任务四竞速路径优化  
原则：先稳定收益，后极限提速；每轮只改一个变量；所有结论以 RAM 日志和实车成功率为准。

## 1. 背景与目标

当前任务四 4 圈耗时约 `22.6s`，已经远低于赛题上限，但任务四按用时计分，仍有继续压缩空间。

本计划目标分三档：

| 档位 | 目标耗时 | 说明 |
|------|----------|------|
| 稳定档 | `21.3s - 21.6s` | 通过低风险参数和控制周期优化获得 |
| 进攻档 | `21.0s - 21.3s` | 加入保守版 C/D 入弯融合 |
| 极限档 | `20.5s - 21.0s` | 弧线提速成功后尝试 |

任务三必须保持可完成，且不得因共享参数调优出现明显退化。

## 2. 当前耗时结构

基于最近一次任务四 RAM 日志，点后动作是主要时间损耗来源。

| 位置 | 动作 | 平均耗时 | 备注 |
|------|------|----------|------|
| AC/BD 直线入 C/D 弯 | `ADVANCE` | 约 `60ms` | 直线点后短距离前推 |
| AC/BD 直线入 C/D 弯 | 红外快转找线 | 约 `330ms` | C/D 入弯主要损耗 |
| CB/DA 弧线出 B/A | `ADVANCE` | 约 `223ms` | 出弧后前推，当前带航向保持 |
| CB/DA 弧线出 B/A | 陀螺转航向 | 约 `207ms` | B/A 转向到下一段直线 |

点后动作合计约 `6.1s/4圈`，是优先优化对象。

需要注意：当前弧线并非完全无风险。日志中曾出现 `lap=1 ph=1` 的长丢线现象，`lost=49`、`maxlost=31`。因此弧线提速应放在后期，而不是第一批参数调整。

## 3. 不变量与约束

- 不修改 `Debug/`、`ti_msp_dl_config.*`、`makefile`、`.out`、`.map` 等生成物。
- 参数优先集中在 `app_config.h`。
- 逻辑改动仅限 `race/race_phase.h`、`race/race_primitives.h`。
- 新增业务逻辑保持 header-only 风格，不新增 `.c` 文件。
- 每次实车测试前确认 `RACE_RAM_LOG_ENABLE` 打开，便于复盘。
- 每次改动后至少执行一次增量编译。

## 4. 评价指标

每次测试记录以下指标：

| 指标 | 通过标准 |
------|----------|
| 任务四完成率 | 当前档位至少连续完成指定次数 |
| 总耗时 | 不高于上一稳定档均值 |
| C/D 入弯 | `sensor_fast_turn` 不得 timeout |
| B/A 出弯 | 转向结束后航向误差不得明显扩大 |
| 弧线循迹 | `maxlost` 不显著高于基线，不能出现连续失控 |
| 强制逻辑 | 不应频繁触发 `FORCE` 路径 |
| 任务三回归 | 阶段完成后任务三至少成功 1 次 |

## 5. 阶段 0：基线确认

目的：建立可比对基线，避免用偶然单次日志做决策。

操作：

1. 使用当前代码跑任务四 `3` 次。
2. 使用当前代码跑任务三 `1` 次。
3. 保存每次 RAM 日志。

记录：

- 任务四总时间。
- 每段 `RACE_SEG` 的 `t0/t1/dist/n`。
- 每个点后 `ADVANCE`、`TURN` 的耗时。
- 弧线段 `lost`、`maxlost`、`endlost`。

通过条件：

- 任务四 3 次均完成。
- C/D 入弯无 timeout。
- 弧线无连续扩大失控趋势。

## 6. 阶段 1：快转控制周期参数化

目的：让任务四 C/D 红外快转使用 `10ms` 控制周期，减少红外停车和转向过冲的量化误差，为后续更激进参数提供余量。

当前问题：

`race_sensor_fast_turn()` 内部硬编码 `CONTROL_PERIOD_MS`，即 `20ms`。任务四主循环和 gyro turn 已经支持 `RACE_TASK4_CONTROL_PERIOD_MS = 10`。

改动：

- 在 `sensor_fast_turn_config_t` 中增加 `control_period_ms` 字段。
- `race_sensor_fast_turn()` 内部使用：

```c
uint32_t control_period_ms =
    (config->control_period_ms != 0U) ?
        config->control_period_ms : CONTROL_PERIOD_MS;
```

- phase 0、phase 2 的任务四入弯配置传入 `RACE_TASK4_CONTROL_PERIOD_MS`。
- 任务三保持默认 `CONTROL_PERIOD_MS`。

验证：

- 编译通过。
- 任务四跑 `3` 次。
- C/D 入弯成功率不下降。
- C/D turn 时间不显著变长。

预期：

直接节省时间较小，但提高后续 P1/P2/L1 的稳定性。

## 7. 阶段 2：advance 参数保守优化

目的：先减少点后直行耗时，但不改变动作结构。

### 7.1 提高 advance PWM

第一档：

```c
#define RACE_POINT_ADVANCE_PWM (340)
```

第二档：

```c
#define RACE_POINT_ADVANCE_PWM (380)
```

注意：`RACE_POINT_ADVANCE_PWM` 同时影响 `RACE_FORCE_FIND_LINE_PWM`，因此要观察失败恢复路径是否过激。

验证：

- 每档任务四跑 `3` 次。
- 无异常后进入下一档。

### 7.2 缩短 C/D 入弯 advance

第一档：

```c
#define RACE_POINT_ADVANCE_COUNT (200)
```

第二档：

```c
#define RACE_POINT_ADVANCE_COUNT (150)
```

验证：

- 每档任务四跑 `3` 次。
- C/D 入弯必须稳定识别弧线。
- 若出现 C/D 快转 timeout 或弧线开头捕获明显变差，回退上一档。

### 7.3 缩短 B/A 出弧 advance

第一档：

```c
#define RACE_ARC_POINT_ADVANCE_COUNT (650)
```

第二档：

```c
#define RACE_ARC_POINT_ADVANCE_COUNT (550)
```

验证：

- 每档任务四跑 `5` 次。
- B/A 后 gyro turn 时间不得明显增加。
- 后续 AC/BD 直线初段航向误差不得明显变差。

阶段 2 完成后：

- 任务三跑 `1` 次，确认共享参数未造成退化。

## 8. 阶段 3：直线末端减速优化

目的：减少 AC/BD 直线末端低速爬行时间。

不建议直接从 `-500/350` 跳到 `-200/100`，按以下三档推进。

保守档：

```c
#define RACE_TASK4_ENTRY_DECEL_START_COUNT (RACE_AC_POINT_ARM_COUNT - 350L)
#define RACE_TASK4_ENTRY_DECEL_RAMP_COUNT  (200L)
```

进攻档：

```c
#define RACE_TASK4_ENTRY_DECEL_START_COUNT (RACE_AC_POINT_ARM_COUNT - 300L)
#define RACE_TASK4_ENTRY_DECEL_RAMP_COUNT  (150L)
```

极限档：

```c
#define RACE_TASK4_ENTRY_DECEL_START_COUNT (RACE_AC_POINT_ARM_COUNT - 200L)
#define RACE_TASK4_ENTRY_DECEL_RAMP_COUNT  (100L)
```

验证：

- 每档任务四跑 `5` 次。
- 观察 AC/BD 点检测距离是否明显后移。
- 观察 C/D 快转是否变长。
- 若快转增加时间抵消直线节省，回退上一档。

退出条件：

- 任一档出现 C/D 快转 timeout、明显打滑、入弯捕线失败，立即回退。

## 9. 阶段 4：C/D 入弯融合

目的：进一步减少 C/D 入弯时 `advance + turn` 分离执行带来的停顿。

不建议直接删除 advance。优先采用保守方案。

### 9.1 保守方案：极短 advance

尝试：

```c
#define RACE_POINT_ADVANCE_COUNT (100)
```

稳定后可尝试：

```c
#define RACE_POINT_ADVANCE_COUNT (80)
```

验证：

- 任务四跑 `10` 次。
- C/D 入弯必须 100% 成功。

### 9.2 进阶方案：快转内最小判停门槛

如果极短 advance 稳定，可考虑去掉 phase 0/2 的 `race_advance_after_point()`，直接进入 `race_sensor_fast_turn()`。

但必须在 `race_sensor_fast_turn()` 内增加最小判停条件，避免刚看到原点线就误停。

可选门槛：

- 最小转向距离，例如 `RACE_ENTRY_TURN_STOP_ARM_COUNT`。
- 最小 yaw 进度，例如 `RACE_ENTRY_TURN_STOP_ARM_CDEG`。
- 两者满足其一后才允许 `line_stop_ready` 生效。

验证：

- 任务四跑 `10` 次。
- C/D 入弯不得 timeout。
- 弧线开头 `first_line_dist_count` 不得明显后移到不可控。
- 若成功率低于 100%，回退到极短 advance 方案。

## 10. 阶段 5：弧线提速

目的：在过渡优化稳定后，再提高 CB/DA 弧线速度。

由于基线日志出现过长丢线，弧线提速必须最后做。

逐档尝试：

```c
#define RACE_TASK4_ARC_BASE_PWM (720)
```

```c
#define RACE_TASK4_ARC_BASE_PWM (740)
```

```c
#define RACE_TASK4_ARC_BASE_PWM (750)
```

只有 `750` 稳定后，才考虑：

```c
#define RACE_TASK4_ARC_BASE_PWM (780)
```

暂不建议直接尝试 `800`。

验证：

- 每档任务四跑 `5` 次。
- 检查 CB/DA 的 `lost`、`maxlost`、`endlost`。
- 若 `maxlost` 明显高于基线，回退上一档。
- 若出现连续丢线或弧线出口点明显提前/滞后，回退上一档。

必要时再微调：

```c
#define RACE_LINE_TURN_DIVISOR
#define RACE_LINE_KD_DIVISOR
#define RACE_LINE_TURN_LIMIT
```

弧线 yaw 辅助 `RACE_ARC_YAW_NAV_ENABLE` 暂不作为第一批优化项。

## 11. 暂缓项目

### 11.1 B/A 出弧 advance 与 gyro turn 重叠

收益存在，但代码复杂度和风险较高。建议在任务四稳定进入 `21s` 左右后再做。

暂缓原因：

- 出弧时车体是否完全越过弧线出口很关键。
- 提前混入 gyro turn 可能导致出口线附近姿态过早偏离。
- 调试需要新增状态机或复合运动原语。

### 11.2 跳过初始对齐

不能简单跳过 `race_task4_align_start_to_ac()`。

原因：

- 当前初始化会先将当前朝向设为 yaw 零点。
- 如果手动把车摆到 AC 方向并跳过对齐，现有 `AC/BD/B/A` 航向目标会整体失配。
- 真要做，需要单独设计“AC 朝向为零”的任务四坐标系和一组目标宏。

因此该项暂缓，不纳入 v1.0 执行。

## 13. 执行顺序总览

| 阶段 | 内容 | 风险 | 预计收益 |
|------|------|------|----------|
| 0 | 基线确认 | 低 | 无，建立基准 |
| 1 | 快转 10ms 控制周期 | 低 | 精度收益 |
| 2 | advance 参数优化 | 低到中 | 约 `0.3s - 0.6s/4圈` |
| 3 | 直线末端减速优化 | 中 | 约 `0.3s - 0.5s/4圈` |
| 4 | C/D 入弯融合 | 中 | 约 `0.2s - 0.4s/4圈` |
| 5 | 弧线提速 | 中到中高 | 约 `0.3s - 0.6s/4圈` |

## 14. 最终建议

正式路线：

```text
基线确认
→ 快转 10ms
→ advance PWM / 距离调优
→ 直线末端减速调优
→ C/D 入弯保守融合
→ 弧线逐档提速
```

不建议路线：

```text
直接删除 C/D advance
直接把弧线 PWM 拉到 780/800
直接跳过初始对齐
一次性同时修改多个宏
```

这份计划以稳定完赛为第一优先级。每个阶段只有在连续实车日志证明稳定后，才进入下一阶段。
