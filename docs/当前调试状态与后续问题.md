# 当前调试状态与后续问题

日期：2026-05-30

本文记录准备 push 前这一版固件的实车状态、关键参数和后续调试重点。当前目标是保留“现在能跑得还可以”的版本，方便后续从同一基线继续调。

## 1. 当前可用状态

- 当前正式任务调度已启用：`ENABLE_CONTEST_TASKS=1`。
- 串口 `UART0` 命令：
  - `01`：任务一 A -> B 直线。
  - `02`：任务二 A -> B -> C -> D -> A。
  - `05`：左右轮编码器/PWM 测试。
  - `00`：强制停车。
- 空闲等待时不再持续输出 `JY62 mode=idle`；收到任务命令后开始输出任务数据，任务结束后停止周期输出。
- 直线段目前表现较好，起步右偏通过抬高右轮 A 的斜坡起点有所缓解。
- 任务二整体已经能跑，但进入弧线的一瞬间仍是主要风险点。

## 3. 当前已知问题

### 3.1 入弧线瞬间容易冲出赛道

现象：

- A -> B 直线后续走线比较直。
- 进入 B -> C 弧线的一瞬间，车可能还没有稳定落到弧线黑线上，容易向外冲出赛道。
- 后续如果红外能正常抓住线，弧线 PD 可以继续工作；问题主要集中在段切换后的最初一小段距离。

当前代码状态：

- 当前版没有保留之前加过的循迹强制转向逻辑。
- 当前版也没有合并其他 branch 里写过的数学前馈逻辑。
- B 点、D 点入线判定按“触发传感器即算入线”的思路处理，没有要求必须经过最中间传感器。

风险判断：

- 如果 `TASK2_AB stop` 后第一条或前几条 `TASK2_BC` 日志中 `raw=0xFF mask=0x00 cnt=0 lost=1`，说明段切换后红外没有及时看到弧线。
- 如果一开始已经看到线，但 `raw_err/err/turn` 很大，说明车头姿态或入弧角度太差，PD 接管太晚。
- 如果起步段 `base` 或左右 PWM 差过大，可能是段切换瞬间速度偏高导致红外还没来得及纠正。

### 3.2 数学前馈逻辑未合并

后续需要从对应 branch 检查数学前馈逻辑，重点确认：

- 是否根据 A -> B 的摆放偏差、入 B 时红外传感器位置或几何关系，预估 B -> C 入弧角度。
- 前馈修正方向是否正确。之前观察到“AB 段摆放偏右，CD 段往外走时更偏右”，需要特别核对符号。
- 前馈只应作为入弧初始辅助，不应长期覆盖红外循迹 PD。
- 合并后要能通过日志看到前馈量，例如 `entry_pos`、`feedforward`、`yaw_adjust` 或类似字段。

### 3.3 不建议直接恢复旧的强制转向

之前的强制转向虽然能帮助找线，但也明显影响循迹性能，容易让车在弧线里被固定转向带偏。

后续如果要加辅助，建议优先考虑：

- 很短时间、很小幅度的入弧预转。
- 只在刚进入 B -> C 或 D -> A 且红外暂时丢线时启用。
- 一旦 `cnt>=1` 立即交还给红外 PD。
- 串口日志必须打印是否处于辅助状态，避免调试时看不出来控制权是谁在主导。

## 4. 明天继续调试建议

1. 先烧录当前版本，确认 `TASK ready` 串口提示包含 `01/02/05/00`。
2. 先用 `05` 跑轮速检测，确认编码器有计数，稳定段大致接近当前 `510/560` 参数。
3. 用 `02` 跑任务二，重点保留这几段日志：
   - `TASK2_AB stop`
   - `TASK2_BC start`
   - `TASK2_BC t=100..500`
   - `TASK2_BC stop`
   - `TASK2_CD t=100..500`
4. 如果入弧第一瞬间 `lost=1`，先降低弧线入口速度或加短时前馈，不要先改直线轮速。
5. 如果入弧能看到线但 `raw_err` 瞬间过大，优先调入弧姿态、前馈方向和 `TASK2_ARC_TURN_SLEW_STEP`。
6. 如果后续弧线能稳定贴线，只把修改限制在“入弧前 0~300ms”附近，避免破坏当前已经能跑的中后段。
